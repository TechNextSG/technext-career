Section 1

System Overview

TechNext Career is a candidate assessment portal: prospective hires take a sequence of psychometric and technical tests, results flow into Supabase, and a live mirror lands in Odoo Recruitment as hr.applicant records with chatter notes.

📝 What candidates do

Visit a role-specific URL, fill candidate info, optionally upload a CV, complete 1–4 short tests, and submit. Results appear instantly.

📊 What recruiters see

Open the admin dashboard or jump straight to Odoo Recruitment. Every applicant has DISC scores, technical-test scores, and chatter notes.

⚙️ How it stays running

Vercel serverless functions, Supabase Postgres + Storage, Odoo SaaS. Zero npm dependencies in serverless handlers — pure Node.

Storage bucket: resumes

  • Public read; writes only via the server-side SUPABASE_SERVICE_ROLE_KEY.
  • File-type allowlist (PDF, DOC, DOCX) enforced by /api/upload-resume — both extension and magic bytes.
  • Size cap: 4.5 MB.
  • Filenames are sanitised and prefixed with a random timestamp to avoid collisions.

Service-role key (server-only)

⚠️ Never expose to the browser

The SUPABASE_SERVICE_ROLE_KEY bypasses Row-Level Security. It must live only in Vercel environment variables and server-side code. If it ever appears in client-side JS, in a public repo, or in a screenshot, rotate it immediately (Supabase Dashboard → Settings → API → "Reset service_role secret").

Odoo — Recruitment Backend

Every test submission upserts to a single hr.applicant record per candidate (matched on email), with a structured chatter note per test and the resume attached on first sync.

What gets written per submission

  1. Upsert hr.applicant by email_from (case-insensitive).
  2. Job link if ODOO_JOB_*_ID is set for the role.
  3. Custom fields on x_tn_* if the optional tn_assessment Odoo module is installed; pre-fetched per warm container and intersected before write.
  4. Chatter note (HTML body) on every submission — always. Universal fallback.
  5. Resume attachment on first sync only — 2 s fetch timeout, 10 MB cap.

Optional companion module

An optional tn_assessment Odoo module adds x_tn_* custom fields and a "TechNext Assessment" page on the applicant form. Source lives in the repo under odoo-module/tn_assessment/. Without it, the integration still works — only the chatter note appears.

Section 7

Authentication & Security

All admin endpoints are protected by an HMAC-signed session token. There is no password on the wire after login.

Login flow

1

Operator types password into the admin login form

Client posts { password } to POST /api/admin-login.

2

Server validates with constant-time compare

crypto.timingSafeEqual against ADMIN_PASSWORD env var. Wrong password → 401. Too many fails from one IP → 429 (8 tries / 5 minutes per warm Lambda).

3

Server issues a signed token

Format: base64url(payload).base64url(hmac-sha256(payload)). Payload contains exp = unix seconds, default 12 h. Signed with ADMIN_SESSION_SECRET.

4

Client stores token in sessionStorage

Every subsequent admin request sends Authorization: Bearer <token>. The dashboard auto-logs-out when the stored expiry passes.

HTTP security headers

HeaderValueScope
Strict-Transport-Securitymax-age=31536000; includeSubDomainsall routes
X-Content-Type-Optionsnosniffall routes
X-Frame-OptionsSAMEORIGINall routes
Referrer-Policystrict-origin-when-cross-originall routes
Permissions-Policycamera=(), microphone=(), geolocation=()all routes
X-Robots-Tagnoindex, nofollow/technext/admin*
Cache-Controlno-store/api/*, admin pages

Input validation

  • /api/submit-test: enforces test_type whitelist, role whitelist, 256 KB payload cap, 500-char string truncation, bounded time_taken_seconds.
  • /api/upload-resume: extension allowlist (PDF/DOC/DOCX) AND magic-byte detection — spoofed extensions cannot smuggle through scripts or images.
  • /api/get-results: token required, query params bounded (limit ≤ 2000, ISO-date regex).

Server-side errors

Responses never include err.message. Operational detail goes to console.error only (visible in Vercel function logs).

Managing Users

Adding a new operator requires touching three to four systems. Walk through each in order.

1. Supabase project access

Supabase has its own user accounts at the project / organisation level. Members can be granted varying roles.

1

Open the project's Team page

Go to Dashboard → Settings → Team (signed in as the current owner).

2

Click "Invite"

Enter the email address of the new operator.

3

Choose a role

Owner — full control including billing. Administrator — manage project + members. Developer — read/write tables and SQL. Read-only — view-only dashboards. For super-admin, pick Owner.

4

Send invite

The user receives an email. Once they accept and create a Supabase account (free), they appear in the team list and can sign in at supabase.com.

About Supabase roles

Supabase auth is for the application's users (e.g. candidates would have their own auth if we used it). What we're managing here is project membership for the operators — that's a separate concept handled in the dashboard's team settings.

2. Vercel project access

1

Open Vercel → team → Members

vercel.com/sky-3677s-projects/~/settings/members

2

Invite by email

Add their email, choose a role:

  • Owner — full control (deployments, env vars, billing)
  • Member — deploy, edit env vars, manage projects
  • Developer — deploy and view; cannot change env vars
  • Viewer — read-only

For super-admin: Owner.

3

Sign-in

They accept the email invite and sign in at vercel.com/login using GitHub or email.

3. GitHub repo access

1

Confirm the user is in the TechNextSG org

Org owner / admin visits github.com/orgs/TechNextSG/people and invites by email or GitHub username.

2

Grant repo-level permission

Open technext-career → Settings → Collaborators and teams → Add people. Choose Admin for full control, Maintain for code + merges, Write for push, Triage for issues, Read for view.

4. Odoo user

1

Settings → Users & Companies → Users

In Odoo at technext.odoo.com, open the Users list and click "New".

2

Fill name and email, set access rights

For super-admin, enable Administration → Settings and Recruitment → Administrator. Save.

3

Send an invitation

Use the "Send Invitation Email" button. The user receives an email to set their password.

5. Admin dashboard access

The admin dashboard uses a shared ADMIN_PASSWORD. To onboard a new operator: share the current password securely (1Password / password manager). To revoke access: change ADMIN_PASSWORD in Vercel env vars and redeploy. All previously issued session tokens remain valid until they expire (default 12 h) but can be force-invalidated by changing ADMIN_SESSION_SECRET.

Section 15

Security Rotations

Required immediately because these secrets were previously in source files committed to Git.

🔄 Required rotations

  1. Supabase service-role JWT — was hardcoded as a fallback in three API files. Reset at Supabase Dashboard → Settings → API → "Reset service_role secret". Then vercel env rm SUPABASE_SERVICE_ROLE_KEY production + vercel env add + redeploy.
  2. GitHub PAT — was in .git/config on the deployment machine. Revoke at github.com/settings/tokens. Generate a new fine-scoped PAT (just repo + workflow) and update local git remote URLs.
  3. ADMIN_PASSWORD — currently set to Admin01! for continuity. Change to something stronger (12+ chars, mixed): vercel env rm ADMIN_PASSWORD production then vercel env add, redeploy, distribute via password manager.

How to rotate cleanly (template)

# 1. Generate or fetch the new secret from the source dashboard
NEW_VALUE="..."

# 2. Remove the old value
vercel env rm SUPABASE_SERVICE_ROLE_KEY production

# 3. Add the new value
echo -n "$NEW_VALUE" | vercel env add SUPABASE_SERVICE_ROLE_KEY production

# 4. Redeploy so the new value is picked up
vercel --prod

# 5. Smoke test
curl -s https://technext-career.vercel.app/api/odoo-ping
Section 16

Operations Runbook

Common tasks an operator may need to perform.

"A candidate submitted but isn't in Odoo"

  1. Open Vercel → project → Logs → filter for evt:odoo_sync_fail.
  2. Hit /api/odoo-ping — confirm "ok": true.
  3. If 503: check the Odoo API key in Vercel env hasn't expired; renew at Odoo → user → Preferences → Account Security.
  4. If 200 but record still missing: the candidate's email may have a typo. Look up the row in Supabase admin and re-run a manual upsert.

"Resume isn't attaching in Odoo"

  1. Confirm candidate_resume_url is set in the Supabase row.
  2. Confirm the Supabase Storage bucket is public-readable.
  3. The chatter note always contains a clickable resume URL even if the file attachment failed.

"Job position isn't linking automatically"

Each role needs an ODOO_JOB_*_ID env var pointing to the corresponding Odoo job position. To find the ID: in Odoo, open Recruitment → Job Positions → the job → look at the URL (e.g. /odoo/recruitment/1 → ID is 1). Set the env var, redeploy.

"I need to lock everyone out immediately"

Rotate ADMIN_SESSION_SECRET (any new value) and redeploy. All in-flight session tokens become invalid instantly.

# generate a new secret + apply
NEW=$(node -e "console.log(require('crypto').randomBytes(48).toString('hex'))")
vercel env rm ADMIN_SESSION_SECRET production
echo -n "$NEW" | vercel env add ADMIN_SESSION_SECRET production
vercel --prod

"Production is down"

  1. Vercel dashboard → project → Deployments. Is the latest deployment "Ready"?
  2. Click into the deployment → Functions → check error rate per endpoint.
  3. Roll back via "Promote to Production" on a previous green deployment.

"Add a new test type"

  1. Create a new HTML file in /technext/<testname>.html following the pattern in iq_test.html.
  2. Add the test_type code to VALID_TEST_TYPES in api/submit-test.js.
  3. Add it to TN_TYPES + TYPE_LABEL + TYPE_COLOR in admin.html.
  4. Optionally add a custom-field mapping in api/_odoo.js (buildCustomFields).
  5. Commit + push + deploy.
